# PRD — 사용자 유입 경로(Acquisition) 트래킹

- 문서 버전: v1.0
- 작성일: 2026-07-22
- 작성/오너: 백엔드 B (AI 오케스트레이션·챗봇 / 운영 도메인 인수)
- 상태: 초안 (구현 대기)
- 관련 파일 사전 조사 완료 — 아래 "영향 범위" 참고

---

## 1. 배경 & 목적

관리자 콘솔([admin_web](admin_web/))에는 회원 수·프리미엄 비율·CS·결제 지표는 있으나,
**"이 사장님이 어떤 경로로 우리 서비스를 알게 됐는가"** 를 알 수 없다.
마케팅 예산 배분, 채널별 전환/리텐션 판단, 추천 프로그램 효과 측정을 위해 유입 경로 데이터가 필요하다.

**핵심 요구:** 데모용 임시 방편이 아니라, **출시 이후에도 코드를 손대지 않고 그대로 실데이터를 수집·표시**할 수 있는 구조로 지금 만든다.
→ 저장 스키마·수집 API·집계 API·관리자 UI를 처음부터 "실데이터 기준"으로 설계하고,
   출시 전까지 실데이터가 없는 구간은 **시딩(backfill)** 으로 채워 화면이 비지 않게 한다.
   실데이터가 들어오기 시작하면 시딩 없이 동일 화면이 그대로 동작한다(인터페이스 불변).

---

## 2. 목표 / 비목표

### 2.1 목표 (In-scope)
- 회원(사장님)의 **가입 시점 유입 채널(first-touch attribution)** 을 1건 저장한다.
- 유입 채널 + 상세값(캠페인/추천코드 등)을 회원 레코드에 영구 보관한다.
- 관리자 대시보드에 **채널별 분포(도넛/막대) + 회원 테이블 유입경로 열**을 표시한다.
- 실데이터가 없는 기존/초기 회원은 **결정적(deterministic) 시딩**으로 자동 보정한다.
- 수집 실패·미전송 시에도 회원가입은 정상 완료된다(트래킹은 절대 가입을 막지 않는다).

### 2.2 비목표 (Out-of-scope, 향후 확장)
- 가입 **이전** 익명 방문자 트래킹(페이지뷰/세션/전환 퍼널) — GA식 이벤트 테이블은 다음 단계.
- 멀티터치 어트리뷰션(여러 접점 가중치 배분). 본 PRD는 first-touch 1건만.
- 광고 매체 API(Meta/Google Ads) 원가·ROAS 연동.
- 채널별 리텐션/코호트 분석 화면.

## 3. 용어: 유입 채널 분류 체계 (Channel Taxonomy)

저장은 **정규 키(canonical key)**, 표시는 **한글 라벨**로 매핑한다. (백엔드 단일 상수로 관리)

| canonical key | 한글 라벨 | 수집 트리거 예시 |
|---|---|---|
| `app_store` | 앱스토어 검색 | 스토어 자연 검색 설치 (UTM 없음, 기본 추정) |
| `web_search` | 포털/구글 검색 | utm_source=google/naver, referrer가 검색엔진 |
| `referral` | 지인 추천 | 추천코드(referral code) 입력, 초대 링크 |
| `instagram` | 인스타그램 | utm_source=instagram (광고/프로필 링크) |
| `youtube` | 유튜브 | utm_source=youtube |
| `naver_blog` | 네이버 블로그/카페 | utm_source=naver_blog/cafe |
| `offline` | 오프라인/전단 | 전단 QR·박람회 코드 |
| `etc` | 기타 | 위에 해당 없음 |
| `unknown` | 미상 | 수집값 없음(구버전 앱/직접 설치) → 시딩 대상 |

- 프론트가 보낸 원문(`utm_source` 등)이 위 키에 매핑 안 되면 서버가 `etc`로 정규화한다.
- 수집값이 아예 없으면 `unknown`으로 저장하고, **집계 시 시딩 규칙으로 표시 채널을 배정**한다(§8).

## 4. 데이터 모델

### 4.1 User 테이블 컬럼 추가 (nullable, 하위호환)
파일: [backend/app/models/user.py](backend/app/models/user.py) — **공동/코어 성격, 백엔드 A와 마이그레이션 협의 필요(§9)**

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| `acquisition_source` | `String(30)` | nullable, default `NULL` | 정규 채널 키 (§3). 미수집 시 NULL |
| `acquisition_detail` | `String(120)` | nullable | 캠페인명/추천코드/utm_campaign 등 원문 보조값 |
| `acquisition_at` | `DateTime(tz)` | nullable | 트래킹 값이 실제 수집된 시각(가입시각과 분리 보관) |

- **모두 nullable** → 기존 로우·구버전 앱과 완전 하위호환. 가입 로직에 영향 없음.
- first-touch 원칙: 값이 이미 있으면 이후 갱신하지 않는다(가입 시 1회 세팅).

### 4.2 마이그레이션
- 운영 DB는 **공유 PostgreSQL** ([backend/.env](backend/.env)). `Base.metadata.create_all`은 **기존 테이블에 컬럼을 추가하지 않으므로** 반드시 마이그레이션이 필요.
- 정식 경로: **alembic revision** 추가 (`alembic/versions/*` = 백엔드 A 소유 → A와 협의/공동 작성).
- 개발 편의(로컬 SQLite 폴백)용: [database.py](backend/app/core/database.py)의 `_ensure_sqlite_schema()` 패턴처럼 컬럼 존재 여부 확인 후 `ALTER TABLE ... ADD COLUMN` 멱등 실행하는 부팅 세이프가드 추가 가능(운영 Postgres는 alembic이 정답).

## 5. 수집(Capture) 설계 — first-touch

### 5.1 소스 판별 우선순위 (프론트 → 서버 전달, 서버가 정규화)
1. **추천코드** 입력값 → `referral` (+ detail=코드)
2. **딥링크/설치 UTM** (`utm_source`, `utm_campaign`) — Expo `Linking`/앱 설치 레퍼러
3. **웹 랜딩 referrer** (웹 가입 경로일 때)
4. 아무것도 없으면 프론트가 `unknown` 전송(또는 미전송) → 서버 `unknown` 저장

### 5.2 API 변경
- `POST /api/v1/auth/signup` ([auth.py:16](backend/app/api/v1/auth.py#L16))
  - `UserCreate` ([schemas/user.py](backend/app/schemas/user.py))에 **optional** 필드 추가:
    - `acquisition_source?: str`, `acquisition_detail?: str`
  - 서버는 값을 §3 규칙으로 정규화 후 저장. **필드가 없어도 기존과 동일하게 가입 성공** (전부 optional).
- 소셜 로그인(Apple 등) 최초 가입 경로도 동일 필드를 함께 넘기면 커버.

### 5.3 프론트(Expo) 작업
- [AuthScreen.tsx](frontend/src/screens/auth/AuthScreen.tsx) 회원가입 폼:
  - (선택) "추천코드" 입력 필드 1개 추가 + 앱 진입 시 딥링크/UTM 파라미터 캐시(AsyncStorage)해 두었다가 가입 payload에 실어 전송.
  - 화면 소유권: auth 화면은 팀 소유 명시가 없어 **프론트 담당과 협의 필요(§9)**.

## 6. 집계(Aggregation) API — 관리자용

신규 엔드포인트 (관리자 콘솔 전용):

```
GET /api/v1/admin/dashboard/acquisition
```

**응답 예시**
```json
{
  "total": 42,
  "seeded_count": 30,          // 실데이터 없어 시딩으로 배정된 수(투명성)
  "channels": [
    { "key": "referral",   "label": "지인 추천",   "count": 14, "ratio": 33.3 },
    { "key": "web_search", "label": "포털/구글 검색", "count": 9,  "ratio": 21.4 },
    { "key": "instagram",  "label": "인스타그램",   "count": 7,  "ratio": 16.7 }
  ]
}
```

- 구현: `db.query(User).all()` → 각 회원의 `acquisition_source`가 있으면 그 값, 없으면(`NULL`/`unknown`) **시딩 규칙(§8)** 으로 채널 배정 → 채널별 count/ratio 산출, 내림차순 정렬.
- 위치: [backend/app/api/v1/admin.py](backend/app/api/v1/admin.py) (기존 `/admin/dashboard/stats` 옆). 채널 상수·정규화·시딩 헬퍼는 별도 함수로 분리.
- (선택) 기존 `GET /admin/users` 응답 각 행에 `"channel": "지인 추천"` 필드 추가 → 회원 테이블에 "유입경로" 열 표시 가능.

## 8. 시딩(Backfill) 규칙 — 출시 전 빈 화면 방지

- 목적: 실데이터가 쌓이기 전에도 대시보드가 그럴듯하게 보이고, 실데이터가 들어오면 자연스럽게 대체.
- 규칙: `acquisition_source`가 NULL/`unknown`인 회원에 대해 **`user.id` 기반 결정적 해시 → 가중 채널 배정**.
  - 결정적이므로 새로고침해도 값이 흔들리지 않음. 저장은 하지 않고 집계 시점에만 배정(원본 오염 없음).
  - 가중치 예: referral 30% / web_search 20% / instagram 15% / app_store 12% / youtube 10% / naver_blog 8% / etc 5%.
- 응답의 `seeded_count`로 "이 중 몇 명이 시딩값인지" 투명하게 노출.
- **실데이터 우선:** 값이 실제로 저장된 회원은 항상 실값 사용. 시딩은 빈 값 보정 전용.
- (운영 전환 옵션) 원한다면 시딩된 값을 DB에 1회 확정 저장하는 관리자 액션도 추후 추가 가능.

## 11. 성공 기준 (Acceptance Criteria)

1. 유입 필드 없이도 `/signup`이 기존과 동일하게 201 성공한다.
2. 유입 필드를 실어 보내면 정규 키로 저장되고 집계에 실값으로 반영된다.
3. 관리자 대시보드에서 채널별 분포와 총계·비율이 보인다(실데이터 0건이어도 시딩으로 렌더).
4. 실데이터가 들어오면 **화면/엔드포인트 코드 변경 없이** 자연스럽게 실값이 표시된다.
5. 트래킹 관련 어떤 실패도 회원가입/로그인을 막지 않는다.

## 13. 향후 확장

- 익명 방문자 이벤트 테이블(`tracking_event`) + 수집 엔드포인트 → 가입 전 퍼널/전환율.
- 채널별 리텐션·LTV 코호트, 추천 프로그램 리워드 정산.
- 광고매체 원가 연동 → 채널별 CAC/ROAS.
- 집계 인터페이스(`/admin/dashboard/acquisition` 응답형)는 그대로 두고 데이터 소스만 교체 → 관리자 UI 재사용.
