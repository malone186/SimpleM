## 0. 배경 — "구현"의 실제 의미

세 기능은 **0→1 신규 개발이 아니다.** 백엔드 C가 만든 naive 스텁이 이미 있고, 특히 **급여정산은 최신 pull(`a39021f`)에서 실제 DB 기반으로 이미 거의 완성**됐다. 따라서 이 작업은 **"naive 스텁 → 실사용 로직으로 고도화 + operation 도메인 인수"** 다.

| 기능 | 현재 상태 (pull 이후) | 실제 남은 일 |
|---|---|---|
| **급여정산** | ✅ 실 DB 계산 완료 (`calculate_payroll` = 실근무시간 기준, `calculate_settlement` = Sale+Expense+인건비) | 버그 2건 픽스 · Expense 입력 API 부재 · 응답 스키마 정합성 |
| **세무** | ⚠️ naive — 단일세율 flat 1개, 입력 수동(body) | 부가세+종소세 근사 로직 · DB 자동집계 |
| **판매예측** | ⚠️ naive — 7일 단순평균+20%, `train_forecast_model.py` 빈 파일, 입력 수동(body) | ARIMA/Prophet 시계열 모델 · DB 자동집계 |

**결정된 방향 (검토 완료):**
1. 소유권 → **operation/ 전체 인수** (C는 다른 작업으로 이동 가정, 진행 전 팀 공지 필요)
2. 판매예측 → **ARIMA/Prophet 시계열 모델**
3. 세무 → **부가세 + 종합소득세 누진 근사**
4. 세무·예측 입력 → **DB 자동집계** (`year_month`만 받으면 Sale/Expense에서 집계)

- CLAUDE.md 상 `models/operation.py`, `schemas/operation.py`, `api/v1/operation.py`, `services/operation/*`, `ml/training/*` 는 원래 **백엔드 C 소유**. 이번에 **B가 전체 인수**.

- 인수 후 CLAUDE.md의 소유권 표를 갱신 (operation → B).

## 4. 기능 2 — 세무계산 (부가세 + 종소세 근사, DB 집계)

`tax_service.py` 전면 재작성. **참고용 근사** — disclaimer 필수 유지.

### 4-1. 부가세 (VAT)
- **일반과세자**: 납부세액 ≈ 매출세액(매출×10%) − 매입세액(비용×10%)
- **간이과세자**: 납부세액 ≈ 공급대가(매출) × 업종부가가치율(음식점업 ≈ 15%) × 10%
- `tax_type: "general" | "simplified"` 파라미터 (기본 general)

### 4-2. 종합소득세 (개인사업자, 누진)
- 과세표준 = 사업소득(매출 − 필요경비) − 기본공제(본인 150만 근사)
- 누진세율표(2024~2026 기준, 산출세액 = 과세표준×세율 − 누진공제):

| 과세표준 | 세율 | 누진공제 |
|---|---|---|
| ~1,400만 | 6% | 0 |
| ~5,000만 | 15% | 126만 |
| ~8,800만 | 24% | 576만 |
| ~1.5억 | 35% | 1,544만 |
| ~3억 | 38% | 1,994만 |
| ~5억 | 40% | 2,594만 |
| ~10억 | 42% | 3,594만 |
| 10억~ | 45% | 6,594만 |

### 4-3. API / 서비스
- `TaxService.estimate_tax(db, year_month, tax_type)` → 내부에서 `get_period_totals`로 매출·비용 자동집계 후 부가세·종소세 각각 계산.
- 엔드포인트: `GET /operation/tax/estimate?year_month=&tax_type=` (DB 자동집계).
  - 하위호환: 기존 `POST /tax/estimate`(body 수동입력)는 유지하거나 deprecated 처리.
- 응답 확장: `TaxEstimateResponse`에 `vat`(부가세), `income_tax`(종소세), `taxable_base`(과세표준), `tax_type` 필드 추가. 요약문·disclaimer 유지.

### 4-4. 검증
- 알려진 값으로 세율 구간별 경계 테스트(1,400만/5,000만 경계), 간이/일반 분기, 매출<비용(과세표준 0) 케이스.

## 5. 기능 3 — 판매예측 (ARIMA/Prophet, DB 집계)

### 5-1. 학습 스크립트 (`ml/training/train_forecast_model.py`, 현재 빈 파일)
- DB에서 일별 매출 시계열 집계(`get_daily_sales_series`) → 시계열 모델 학습 → 아티팩트 저장.
- 엔진: **statsmodels ARIMA 기본** / Prophet 선택.
- 저장: `backend/app/ml/artifacts/forecast_<store_id>.pkl` (모델 or 파라미터).
- CLI 실행 가능: `python -m app.ml.training.train_forecast_model --store <id>`.

### 5-2. 예측 서비스 (`forecasting_service.py` 재작성)
- `forecast_sales(db, target_date, store_id, has_event=False, engine="arima")`:
  1. DB에서 최근 시계열 로드 (자동집계)
  2. 저장된 모델 로드 (없으면 즉석 학습)
  3. `target_date`까지 예측 → `predicted_sales`, `predicted_quantity`
  4. **폴백**: 데이터 부족(<14일) 또는 모델 오류 시 → 기존 이동평균 로직으로 graceful fallback
  5. `evidence_summary`에 사용 엔진·근거 명시, 참고용 disclaimer
- 이벤트 보정(+20%)은 옵션으로 유지.

### 5-3. API / 도구
- `POST /operation/forecast/sales` → body에 `sales_data` 없으면 DB 자동집계로 예측(둘 다 지원 or DB 우선). `store_id`·`target_date` 기반.
- `forecasting_tools.py`(챗봇 도구)도 인수 파일 → DB 집계 시그니처에 맞게 갱신.

### 5-4. 검증
- 합성 시계열(주기성 있는 30~60일)로 예측값 sanity 체크, 데이터 부족 시 폴백 동작 확인, 모델 아티팩트 저장/로드 왕복.

## 8. 리스크 / 주의

- **머지 충돌**: operation은 C의 활성 영역 → 인수 합의·타이밍이 최우선.
- **Prophet 설치**: Windows에서 까다로움 → 기본 ARIMA, Prophet은 optional로.
- **데이터 부족**: 신규 매장은 시계열이 짧음 → 예측은 반드시 이동평균 폴백 유지.
- **세무 정확도**: 어디까지나 **참고용 근사** — 공제·업종·과세유형 단순화. disclaimer 필수.
- **컬럼 추가 시**: alembic 없이 create_all이라 기존 dev DB는 재생성 필요할 수 있음(이번 범위는 신규 테이블/읽기 위주라 영향 적음).
