| 항목 | 내용 |
|---|---|
| 문서 버전 | v1.0 (2026-07-14) |
| 팀 구성 | 백엔드 3명(A·B·C) + 프론트 2명(A·B) |
| 저장소 | github.com/malone186/SimpleM |

---

## 1. 개요

### 1.1 제품 정의
카페 사장님을 위한 **운영 통합 플랫폼**. 재료 등록부터 판매·재고·원가·발주·세무까지의 운영 데이터를 한곳에 모으고, 그 위에 AI(리포트·OCR·예측·챗봇)를 얹어 "숫자를 말로 바꿔주는" 경영 파트너를 제공한다.

### 1.2 문제 정의
- 소규모 카페는 재고·원가·발주를 수기/엑셀로 관리해 원가율 변동, 재고 누수를 뒤늦게 발견한다.
- POS 매출 데이터는 있지만 해석해 줄 사람이 없다 (숫자는 있는데 인사이트가 없음).
- 거래명세서 입력, 발주 비교, 스케줄 짜기 같은 반복 업무에 시간을 뺏긴다.

### 1.3 해결 방향
1. **ERP 기본기**: 재료→메뉴→재고→판매→원가로 이어지는 데이터 파이프라인을 자동화
2. **AI 레이어**: 축적된 데이터를 RAG로 활용해 리포트·예측·감지·추천을 제공
3. **챗봇 단일 창구**: 단순 조회·시각화·확정 액션만 전용 화면으로 두고, 나머지 기능은 챗봇 하나로 통합 (프론트 화면 5개로 압축)

### 1.4 핵심 원칙
- 로직(`*_service.py`)과 챗봇 등록(`*_tools.py`)을 분리 — 팀원 간 파일 충돌 최소화
- **돈이 걸린 액션(발주/지급/신고)은 AI가 실행하지 않는다.** `propose_`/`draft_` 접두어로 초안만 만들고, 확정은 반드시 사람이 전용 화면에서 승인

## 4. 핵심 워크플로우 (데이터 파이프라인)

```
재료 등록 → 메뉴 등록(레시피=재료 구성) → 초기 재고 입력
   → 판매 (POS 연동 또는 수동 입력)
   → 재고 자동 차감 (레시피 기준)
   → 원가 자동 계산 → 원가율 분석
   → 재고 부족 알림 → 발주 추천(초안) → [사람이 승인]
   → 매출·이익 분석 → AI 주간 경영 리포트
```

이 파이프라인이 모든 AI 기능의 데이터 원천이다. **ERP 기본기(재료~원가)가 MVP의 최우선 순위인 이유.**

## 5. 기능 요구사항

우선순위: **P0** = MVP 필수 / **P1** = MVP 직후 / **P2** = 차기

### 5.1 ERP 기본 (P0)

| ID | 기능 | 설명 | 담당 |
|---|---|---|---|
| ERP-1 | 회원/로그인 | Firebase 인증, 매장 단위 계정 | 백엔드 A |
| ERP-2 | 재료 관리 | 재료 등록(단위, 단가), 단가 이력 관리 | 백엔드 A |
| ERP-3 | 메뉴 관리 | 메뉴 등록 + 레시피(재료·용량 구성) | 백엔드 A |
| ERP-4 | 재고 관리 | 초기 재고 입력, 판매 시 레시피 기준 자동 차감, 입고 반영 | 백엔드 A |
| ERP-5 | 판매 입력 | POS 연동(가능 시) 또는 수동 입력 | 백엔드 A |
| ERP-6 | 원가 계산 | 메뉴별 원가·원가율 자동 계산, 단가 변동 반영 | 백엔드 A |
| ERP-7 | 재고 부족 알림 | 안전재고 미달 시 알림 | 백엔드 A |
| ERP-8 | 발주 관리 | 발주 초안 생성(AI 추천 연동) → 사장 승인 → 발주 확정 | 백엔드 A + 프론트 A |
| ERP-9 | 스케줄·급여·정산 | 알바 스케줄 편집, 급여 계산, 정산 | 백엔드 C + 프론트 B |
| ERP-10 | 세금 관리 | 세금 계산, 신고 초안(draft_) | 백엔드 C |
| ERP-11 | 법령 검색 | 법령 크롤링 + RAG 검색 (챗봇으로 제공) | 백엔드 C(수집)/B(챗봇) |
| ERP-12 | 서류 자동화 | 문서 생성 자동화 (챗봇으로 제공) | 백엔드 B |

**수용 기준(핵심)**: 판매 1건 입력 시 → 레시피 기준 재고 차감 + 원가 반영이 트랜잭션으로 일관되게 처리된다. 재고가 음수가 되는 경우 경고를 남긴다(차감은 허용, 누수 감지의 신호로 사용).

### 5.2 AI 기능

| ID | 기능 | P | 담당 |
|---|---|---|---|
| AI-1 | 주간 경영 리포트 | P0 | 백엔드 B |
| AI-2 | 명세서/영수증 OCR 자동 입고 | P0 | 백엔드 B |
| AI-3 | 판매량 예측 + 발주 추천 | P1 | 백엔드 C |
| AI-4 | 시간대 분석 기반 알바 스케줄 추천 | P1 | 백엔드 C |
| AI-5 | 재고 누수 자동 감지 | P1 | 백엔드 C(로직)/B(리포트 연동) |
| AI-6 | 로스터리 원두 비교 | P2 | 백엔드 C(크롤링)/B(챗봇) |
| AI-7 | 원두 추천 + 리뷰 데이터 축적 | P2 | 백엔드 C/B |

**AI-1. 주간 경영 리포트** — 숫자 → 말
- 입력: 주간 매출·이익·원가율·재고 데이터 (RAG로 우리 DB 조회)
- 출력: 자연어 리포트. 예: *"이번 주 원가율이 3%p 올랐는데, 우유 단가 인상이 원인이에요. 라떼 계열 마진이 특히 눌리고 있으니 가격 조정을 검토해 보세요."*
- 요건: 변동의 **원인**(어떤 재료 단가, 어떤 메뉴군)까지 짚을 것. 근거 숫자를 함께 표시해 검증 가능하게. 매주 자동 생성 + 챗봇에서 조회.

**AI-2. OCR 자동 입고**
- 입력: 거래명세서/영수증 사진 → 출력: {상품, 단가, 수량} 쌍 → 재고 시스템 등록 초안
- 구현 후보 2가지를 비교 테스트 후 채택:
  - ① VLM 단독: 이미지에서 바로 상품·가격·수량 쌍 추출 (deepseek-ocr, qwen 계열 등)
  - ② OCR + LLM 2단계: OCR로 전체 텍스트 추출 후 LLM이 쌍으로 구조화 (paddle OCR은 key-value 쌍 인식이 안 되므로 반드시 LLM 후처리 필요)
- 요건: 인식 결과는 **등록 초안**으로 사용자에게 보여주고 수정·확인 후 재고 반영 (자동 확정 금지)

**AI-3. 판매량 예측 + 발주 추천**
- 입력 피처: 매장 위치, 날씨, 요일, 주변 행사, POS 판매 이력
- 출력: 익일/금주 예상 판매량 → 레시피 역산으로 재료 소요량 → 발주 추천 초안(`propose_order`)
- 제약: **최소 n일 이상 판매 데이터가 쌓인 사용자에게만 예측 제공** (n은 실험으로 결정, 초기 가설 30일)
- 접근: 시계열 모델(ARIMA/Prophet) 단순 예측 → 이벤트 정보로 보정(예: 행사일 +20% 상향) → 데이터 쌓이면 xgboost/lightgbm 피처 모델로 고도화
- 콜드스타트: Kaggle 공개 데이터셋으로 초기 모델 학습 (우리 서비스 인풋과 유사한 것 선택)
  - coffee-shop-sales-dataset (xavierberge)
  - coffee-shop-daily-revenue-prediction-dataset (himelsarder)
  - coffee-sales-dataset (navjotkaushal)

**AI-4. 알바 스케줄 추천**
- 매출·이익을 시간대별 분석 → 피크/한산 시간 도출 → 스케줄 추천 초안
- 확정·편집은 프론트 B의 스케줄/급여 화면에서 사람이 수행

**AI-5. 재고 누수 자동 감지**
- 기본 원두 세팅값(레시피 도징량) 대비 실제 소비량을 **매일** 비교
- 판매량과 소비량 불일치 시 원인 후보까지 제시: *"도징량 점검 필요"*, 폐기 누락, 레시피 오차 등
- 감지 결과는 알림 + 주간 리포트(AI-1)에 반영

**AI-6. 로스터리 원두 비교**
- 여러 로스터리 사이트 크롤링 → 단가, 로스팅 강도, 맛 프로필, 원산지 통합 비교
- 제공 채널: 챗봇 ("에티오피아 미디엄 로스팅 원두 kg당 가격 비교해줘")

**AI-7. 원두 추천 + 리뷰 축적**
- 카페 니즈(맛 방향, 가격대) 기반 원두 추천
- 원두별 사용 데이터 축적: *"이 원두는 다른 카페에서도 도징 편차가 크다는 리뷰가 있어요"*
- 제공 채널: 챗봇

### 5.3 AI 챗봇 (통합 창구, P0) — 백엔드 B + 프론트 B

- **아키텍처**: 메인 에이전트(`services/ai/agents/main_agent.py`)가 기능별 도구/서브에이전트를 오케스트레이션. LangChain Agent 또는 LangGraph(커스텀 워크플로우 필요 시).
- **도구 등록**: A·C가 각자 `*_tools.py`에 `@tool` 래퍼 작성 → `tool_registry.py`에 import 한 줄 추가(알파벳순) → 챗봇이 전체 기능을 자동 획득
- **다루는 범위**: 리포트 조회, 재고/매출 질의, 발주 초안 생성, 법령 검색, 문서 생성, 원두 비교·추천 등 — 전용 화면이 없는 모든 기능
- **안전장치**: 돈이 걸린 도구는 `propose_`/`draft_`만 노출. 확정 도구는 챗봇에 등록하지 않는다.

## 6. 화면 구성 (React Native)

### 하단 탭 (6개)

| 화면 | 내용 | 담당 |
|---|---|---|
| ① 대시보드 | 원가율·재고·매출 시각화 | 프론트 A |
| ② 재고 | 재고 조회, OCR 입고 확인 | 프론트 A |
| ③ 발주 승인 | 챗봇/AI가 만든 발주 초안 검토·승인 | 프론트 A |
| ④ 챗봇 | 통합 챗봇 (리포트·추천·법령·문서·비교 전부) | 프론트 B |
| ⑤ 운영 | 스케줄/급여 편집, 세금 확인 | 프론트 B |
| ⑥ 관리 | ERP 관리 허브 — 아래 세부 화면으로 진입 | 프론트 A/B |

### 관리 허브 세부 화면 (스택 진입)

| 화면 | 기능 | PRD |
|---|---|---|
| 재료 관리 | 재료 등록, 단위·단가, 단가 이력 | ERP-2 |
| 메뉴 관리 | 메뉴 등록 + 레시피(재료 구성), 원가율 | ERP-3 |
| 판매 입력 | POS 연동/수동 판매 입력 → 재고 자동 차감 | ERP-5 |
| 원가 분석 | 메뉴별 원가·원가율, 단가 변동 반영 (숫자 화면 — 브루 미노출) | ERP-6 |
| 법령 검색 | 법령 RAG 검색 (챗봇과 동일 백엔드) | ERP-11 |
| 서류 자동화 | 문서 템플릿 → 초안 생성(draft_) | ERP-12 |
| 세금 관리 | 부가세·원천세 (운영 탭과 공유) | ERP-10 |
| 프로필 | 상호·사진·비밀번호 수정, 로그아웃 | ERP-1 |

> 화면 압축 원칙(단순 조회·확정 액션만 전용 화면, 나머지는 챗봇)은 유지하되, ERP 기본기(재료~원가)는 사장님이 직접 관리하는 빈도가 높아 전용 화면으로 승격.

공동 소유: `RootNavigator`, `lib/api/client.ts` (수정 시 팀 공지), `types/api.d.ts` (CI 자동 생성, 직접 수정 금지)

## 7. 비기능 요구사항

- **비용 통제**: Gemini API 사용한도(일/월 예산 상한) 설정 필수. LLM 호출은 단일 모듈로 추상화해 모델·프로바이더 교체가 한 곳 수정으로 가능하게 (가격 비교 후 최저가부터 테스트하는 전략 지원)
- **안전성**: 금전 액션은 초안-승인 2단계 강제. AI 출력에는 근거 데이터 첨부
- **인증/보안**: Firebase 인증, 매장 단위 데이터 격리. API 키는 환경변수(.env)로 관리, 저장소 커밋 금지
- **가용성**: LLM API 장애 시 ERP 기본 기능(재고·판매·원가)은 독립적으로 동작해야 함
- **예측 품질**: 데이터 부족 사용자에게는 예측을 숨기고 "데이터 수집 중 (D-n)" 표시

## 8. 데이터 모델 개요 (초안)

- **inventory 도메인 (A)**: Ingredient(재료, 단가 이력), Menu, Recipe(메뉴-재료 매핑), Stock, StockTransaction(입고/차감/조정), Sale, Order(발주)
- **ai 도메인 (B)**: Report(주간 리포트), OcrDocument(원본 이미지·인식 결과·확정 여부), ChatSession/ChatMessage, DocumentTemplate
- **operation 도메인 (C)**: Staff, Schedule, Payroll, Forecast(예측 결과), LeakageAlert(누수 감지), RoasteryBean(크롤링 원두), TaxRecord

각 도메인은 `models/{inventory,ai,operation}.py`에 1파일씩. `models/__init__.py` 등록은 알파벳순.

## 9. 마일스톤

| 단계 | 범위 | 완료 기준 |

| **M4: 원두 생태계** | AI-6, AI-7 | 챗봇에서 로스터리 비교·원두 추천 제공 |

## 10. 리스크 & 오픈 이슈

| 리스크 | 대응 |
|---|---|
| POS 연동 불확실 (업체별 API 상이) | MVP는 수동 입력 우선, POS 연동은 별도 조사 |

| 크롤링 대상 사이트 구조 변경/약관 | 로스터리별 어댑터 분리, robots·약관 확인 후 진행 |
| 프론트 스택 표기 불일치 (현재 골격은 Next.js식 `page.tsx`) | React Native 확정 시 `frontend/` 구조를 RN 프로젝트로 재생성 필요 — 팀 결정 사항 |
